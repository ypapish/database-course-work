1. Специфікація CreateCategory

ID:CreateCategory
НАЗВА:Створити категорію медіаконтенту.
УЧАСНИКИ:Адміністратор, Система.
ПЕРЕДУМОВИ:Немає.
РЕЗУЛЬТАТ:Створена категорія медіаконтенту.
ВИКЛЮЧНІ СИТУАЦІЇ:Ex.001 Невалідні дані.
ОСНОВНИЙ СЦЕНАРІЙ:1. Адміністратор надає системі дані, необхідні для створення категорії.
2. Система перевіряє валіднасть наведених даних та зберігає категорію (можлива EX.001).
3. Система повідомляє адміністратору про успішність виконання операції.

2. Специфікація CreateCategory

ID:AnalyzeSentiment
НАЗВА:Проаналізувати ресурси для отримання тональності контенту.
УЧАСНИКИ:Користувач, Система.
ПЕРЕДУМОВИ:Існувння хоча б одного ресурсу.
РЕЗУЛЬТАТ:Тональність контенту.
ВИКЛЮЧНІ СИТУАЦІЇ:Ex.001 Невалідні дані.
ОСНОВНИЙ СЦЕНАРІЙ:1. Адміністратор надає системі дані, необхідні для аналізу.
2. Система перевіряє валіднасть наведених даних та аналізує ресурси (можлива EX.001).
3. Система надає результат користувачеві.

3. Специфікація AddResource

ID:AddSource
НАЗВА:Додати джерело для аналізу даних.
УЧАСНИКИ:Адміністратор, Система.
ПЕРЕДУМОВИ:Немає.
РЕЗУЛЬТАТ:Новий джерело аналізу данних.
ВИКЛЮЧНІ СИТУАЦІЇ:EX.001 Джерело недоступне.
ОСНОВНИЙ СЦЕНАРІЙ:1. Адміністратор надає системі дані, за якими вона може отримати інформацію.
2. Система перевіряє валідність наведених данних (можлива EX.001).
3. Система зберігає джерело.
4. Система повідомляє адміністратору про успішність виконання операції.

4. Специфікація UpdateSource

ID:UpdateSource
НАЗВА:Змінити відомості про джерело аналізу даних.
УЧАСНИКИ:Адміністратор, Система.
ПЕРЕДУМОВИ:Існування джерела аналізу даних.
РЕЗУЛЬТАТ:Змінений ресур джерела данних.
ВИКЛЮЧНІ СИТУАЦІЇ:EX.001 Джерело недоступне.
EX.002 Джерело відсутнє в системі.
ОСНОВНИЙ СЦЕНАРІЙ:1. Адміністратор надає системі дані, за якими вона може ідентифікувати джерело.
2. Система перевіряє валіднасть джерела та ідентифікує його (можлива EX.001 та EX.002).
3. Адміністратор надає дані, на які потрібно замініти джерело.
4. Система змінює джерело відповідно до доданих даних.
5. Система повідомляє адміністратору про успішність виконання операції.

5. Специфікація DeleteSource

ID:DeleteSource
НАЗВА:Видалити джерело аналізу даних.
УЧАСНИКИ:Адміністратор, Система.
ПЕРЕДУМОВИ:Відсутність існування джерела аналізу даних.
РЕЗУЛЬТАТ:Нове джерело аналізу даних.
ВИКЛЮЧНІ СИТУАЦІЇ:EX.001 Джерело відсутнє в системі.
ОСНОВНИЙ СЦЕНАРІЙ:1. Адміністратор надає системі дані, за якими вона може отримати інформацію.
2. Система перевіряє нявність джерела (можлива EX.001).
3. Система видаляє інформацію про джерело.
4. Система повідомляє адміністратору про успішність виконання операції.

6. Специфікація GetData

ID:GetData
НАЗВА:Отрмати проаналізовану інформацію.
УЧАСНИКИ:Користувач, Система.
ПЕРЕДУМОВИ:Наявність хоча б одного ресурсу аналізу даних.
РЕЗУЛЬТАТ:Проаналізована інйормація.
ВИКЛЮЧНІ СИТУАЦІЇ:EX.001 Невалідні дані.
ОСНОВНИЙ СЦЕНАРІЙ:1. Адміністратор робить запит на отримання даних.
2. Система аналізує дані.
3. Система надає результат користувачеві.

7. Специфікація ExportData

ID:ExportData
НАЗВА:Експортувати дані із системи.
УЧАСНИКИ:Користувач, Система.
ПЕРЕДУМОВИ:Наявність хоча б одного ресурсу аналізу даних та прав доступу до експорту.
РЕЗУЛЬТАТ:Успішний експорт даних.
ВИКЛЮЧНІ СИТУАЦІЇ:EX.001 Помилка експорту даних.
EX.002 Недостатньо прав для виконання експорту.
ОСНОВНИЙ СЦЕНАРІЙ:1. Користувач ініціює запит на експорт даних, вказуючи бажаний формат.
2. Система перевіряє доступ до ресурсів і прав користувача.
3. Система експортує дані у вказаний формат.
4. Система надає користувачеві повідомлення про успішність операції (або помилку в разі EX.001 або EX.002).

8. Специфікація ManageUsers

ID:ManageUsers
НАЗВА:Керувати обліковими записами користувачів.
УЧАСНИКИ:Адміністратор, Система.
ПЕРЕДУМОВИ:Наявність доступу до інтерфейсу керування обліковими записами користувачів у адміністратора.
РЕЗУЛЬТАТ:Оновлений список користувачів або зміни в їхньому статусі.
ВИКЛЮЧНІ СИТУАЦІЇ:EX.001 Невалідні дані.
EX.002 Користувача не знайдено.
ОСНОВНИЙ СЦЕНАРІЙ:1. Адміністратор ініціює дію для керування обліковими записами користувачів.
2. Система перевіряє валідність даних (можлива EX.001).
3. Система здійснює відповідні зміни в базі даних облікових записів.
4. Система повідомляє адміністратору про успішність виконання операції або про помилку (можлива EX.002).

9. Специфікація LogIn

ID:LogIn
НАЗВА:Авторизація в систему.
УЧАСНИКИ:Користувач, Система.
ПЕРЕДУМОВИ:Наявність облікового запису в системі.
РЕЗУЛЬТАТ:Успішна авторизація.
ВИКЛЮЧНІ СИТУАЦІЇ:EX.001 Неправильні дані для авторизації.
EX.002 Обліковий запис не знайдений.
ОСНОВНИЙ СЦЕНАРІЙ:1. Користувач надає свої дані для авторизації (логін і пароль).
2. Система перевіряє наявність облікового запису та валідність введених даних (можлива EX.001, EX.002).
3. Система надає користувачеві доступ до персоналізованої інформації та ресурсів.

10. Специфікація SignUp

ID:SignUp
НАЗВА:Зареєструвати новий обліковий запис.
УЧАСНИКИ:Користувач, Система.
ПЕРЕДУМОВИ:Відсутність облікового запису користувача.
РЕЗУЛЬТАТ:Новий обліковий запис.
ВИКЛЮЧНІ СИТУАЦІЇ:EX.001 Невалідні дані.
EX.002 Користувач уже зареєстрований.
ОСНОВНИЙ СЦЕНАРІЙ:1. Користувач надає необхідні дані для реєстрації.
2. Система перевіряє валідність введених даних (можлива EX.001).
3. Система перевіряє, чи є вже такий користувач(можлива EX.002).
4. Система створює новий обліковий запис користувача.
5. Система повідомляє користувачу про успішність реєстрації.
